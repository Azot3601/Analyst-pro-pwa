# Архитектура обучения: тренажёр системного аналитика

Документ фиксирует целевую педагогику тренажёра и принципы, по которым строятся новые
модули. Первый реализованный по этим принципам модуль — «Анализ требований: Элиситация →
User Stories» (вкладка **Требования** в `/trainer`).

## Зачем менять подход

Текущий контент силён по **инструментам** аналитика (SQL — 25 уроков, API/контракты —
~31 задача), но почти не тренирует **ядро профессии**: цикл
«мутный запрос стейкхолдера → правильные вопросы → модели → спецификация →
дизайн контракта → валидация и trade-off».

Две системные проблемы прежнего тренажёра:

1. **Тренируется не та мышца.** ~90% — синтаксис SQL/API. Новичку непонятно «зачем»,
   а навык не растёт вширь до уровня настоящей аналитической работы.
2. **Нет удержания вдолгую.** Решённый урок больше не всплывает: нет интервального
   повторения, нет разбора *почему* ошибся, нет синтезирующих капстоунов.

## Спиральная программа вокруг реального цикла аналитика

Обучение строится слоями; один и тот же цикл повторяется на всё более сложных кейсах.

### Слой 0 — Фундамент (узнавание / синтаксис)
Текущие SQL / REST / JSON / OpenAPI / Интеграции. **Оставляем как есть** — это база
инструментов.

### Слой 1 — «Петля аналитика»
Пять шагов, прогоняемых на каждом кейсе:

1. **Элиситация** — из сырого брифа/интервью вытащить требования, отделить факты от
   допущений, найти неоднозначности, сгенерировать уточняющие вопросы.
2. **Моделирование** — context diagram, BPMN, ERD, sequence/state, use-case.
3. **Спецификация** — user story + критерии приёмки (Gherkin), НФТ, edge/негативные кейсы.
4. **Дизайн контракта** — сюда встаёт существующий API-квест (REST/JSON/OpenAPI/интеграции).
5. **Валидация и trade-off** — ревью на дыры, ADR, матрица рисков.

> Реализовано в этой итерации: шаги **1** и **3** в модуле «Требования» (PoC петли).

### Слой 2 — Капстоуны
Сквозные кейсы на одном домене (напр. «процесс возврата/рефанда для маркетплейса»),
собирающие 3–5 концепций из слоёв 0–1 и оцениваемые целостно.

## Движок удержания (принципы на будущее)

- **Интервальное повторение** по микроконцептам (а не по урокам целиком): решённое
  переспрашивается через 1д / 3д / 1нед, управление кривой забывания.
- **Адаптивная обратная связь по типу ошибки**: ошибку классифицируем и поднимаем
  связанный концепт, а не показываем просто ✅/❌.
- **Worked examples + рефлексия**: после попытки раскрываем экспертное решение и просим
  сравнить со своим. Сильнее всего качает перенос навыка.

## Принцип офлайн-оценки открытых задач (текущий выбор)

Приложение полностью клиентское и офлайн. Открытые навыки (требования, модели, спеки)
по своей природе неоднозначны, но мы оцениваем их **детерминированно, без LLM**, через:

- **структурированные конструкторы** вместо свободного текста (выбор/разметка/Gherkin
  из набора клауз);
- **ключ-эталон с нормализацией** (регистр, синонимы) — как в `checkRest`;
- **чек-листы с дистракторами** (нужное подмножество присутствует, «шумовое» отсутствует) —
  как в `checkIntegration`.

Технически переиспользуется существующий контракт диагностик
`ApiQuestDiagnostic[]` / `ApiQuestCheckResult` из
[`src/shared/lib/apiQuestCheckers.ts`](../src/shared/lib/apiQuestCheckers.ts).

LLM-оценка по рубрике (serverless на Vercel) — возможное расширение, **сейчас вне объёма**.

## Дорожная карта (после PoC)

1. Шаги 2 и 5 петли: лёгкие конструкторы моделей (context/sequence как структурный
   выбор) и ADR/риски как чек-лист.
2. Капстоун, связывающий «Требования → Контракт (API-квест)» на одном домене.
3. Движок удержания: расписание повторений поверх `solvedTaskIds` + слабых зон.
4. (Опц.) LLM-оценщик открытых ответов по рубрике как serverless-функция.
