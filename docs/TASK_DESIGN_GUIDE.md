# TASK_DESIGN_GUIDE

## Структура задачи

Каждая задача содержит:

- бизнес-контекст;
- зачем это аналитику;
- исходные артефакты;
- что нужно сделать;
- критерий правильности;
- многоуровневые подсказки;
- объяснение;
- частые ошибки;
- связанные знания.

## Подсказки

1. Лёгкий намёк.
2. Направление решения.
3. Структурная подсказка.
4. Почти готовое решение.
5. Разбор после ответа.

## SQL Quest

SQL-задачи проектируются по формуле:

`сюжетная причина → SQL-действие → проверяемый результат → ошибка с объяснением → связанное знание → награда`

Правила:

- SQL и аналитическое мышление важнее сюжета.
- Сюжет объясняет, зачем нужен запрос, но не заменяет бизнес-контекст.
- `trial` проверяет точность концепта.
- `case` добавляет риск неверной трактовки, ловушку данных или интеграционный смысл.
- Ошибка должна объяснять разницу по строкам, колонкам и вероятной причине.
- Каждая задача должна вести к knowledge node, который помогает исправить ошибку.

## API Contract Quest

Формула API-задачи: бизнес-процесс → источник/потребитель → часть контракта → структурированное действие → rule-based проверка → диагностика → knowledge link.

Одна задача тренирует один главный навык. REST проверяет request и response; JSON — payload против schema; OpenAPI — части договора; Integration — устойчивость обмена; SOAP — XML-структуру и Fault.

REST-урок дополнительно содержит:

- `Что сейчас тренируем` — конкретную часть request/response;
- `Где это в жизни` — рабочую ситуацию и смысл для аналитика;
- пример path parameter value, если endpoint содержит placeholder;
- computed URL, формируемый из template и введённых значений.

Нулевой вводный урок не оценивается и не влияет на прогресс. Он объясняет anatomy request до первой проверяемой задачи.

## REST Beginner Principle

Каждая новая сущность объясняется до обязательного ввода. Порядок REST-урока:

`объяснение → пример → мини-практика → самостоятельная задача → диагностика → объяснение результата`.

Перед рабочей областью обязательно показываются:

- конкретный навык текущего шага;
- объяснение простыми словами;
- практическое применение для системного аналитика;
- список заполняемых частей request;
- ожидаемые status code и response fields;
- не изменяющий draft пример заполнения;
- явные тестовые значения.

Для JSON body нужен contract panel: required/optional, типы, enum, nullable и корректный пример. Ошибки request группируются по первичной причине; status и отсутствующие response fields описываются как последствия.
