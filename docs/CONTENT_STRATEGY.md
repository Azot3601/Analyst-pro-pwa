# CONTENT_STRATEGY

Контент строится вокруг практики системного аналитика: бизнес-контекст, артефакты, критерий правильности, подсказки, объяснение, частые ошибки и связь с базой знаний.

## Минимальный полный объём

- 30 SQL-задач.
- 25 REST/API-задач.
- 25 JSON/OpenAPI-задач.
- 15 интеграционных кейсов.
- 80+ узлов базы знаний.
- 20+ инструментов и чеклистов.

## Подход

- Сначала качественная репрезентативная часть по всем доменам.
- Затем расширение через однотипные типизированные data files.
- Каждый материал должен быть валидируемым и связанным с knowledge graph.

## SQL-траектория

SQL больше не должен выглядеть как набор независимых карточек. Базовый поток строится как SQL Quest Mode, подробнее в `docs/SQL_QUEST_MODE.md`.

Текущий объём SQL Quest:

- 8 глав;
- 24 задачи;
- два пути: `trial` и `case`;
- 10 точных SQL knowledge links;
- XP/ranks и persistent progress.

Каждый шаг обязан показывать таблицы, связи, бизнес-контекст, зачем это аналитику, критерии успеха, подсказки, табличный результат, частые ошибки и диагностический разбор mismatch.

Backlog:

- расширить реальные кейсы миграциями, сверкой API/БД и просадкой конверсии.

## API Contract Quest

Шаблонная генерация API/JSON/Integration контента заменена на 28 самостоятельных задач. Траектория идёт от anatomy REST request к JSON Schema, OpenAPI, устойчивым интеграциям и базовому чтению SOAP. Детали: docs/API_CONTRACT_QUEST.md.

### Beginner REST Ladder

Первые восемь REST-задач образуют самостоятельный маршрут с нуля:

1. GET конкретного ресурса и path parameter.
2. POST, Content-Type и JSON body.
3. Query params, фильтрация и pagination.
4. PATCH и частичное обновление.
5. DELETE черновика и HTTP 204 без body; бизнес-отмена отделена от удаления.
6. Idempotency-Key и повтор платежа.
7. PUT, If-Match и защита от потерянного обновления.
8. HTTP 422 и error model.

Каждый пример выполняется локальным simulator и обязан проходить тем же checker, что пользовательский ответ. Тестовые идентификаторы не нужно угадывать: `ORD-1001`, `CUST-1001`, `PRD-10`, `PAY-1001`, `pay-001`.

Общий anatomy-onboarding показывается только перед первой REST-задачей. Начиная со второй задачи остаётся компактный блок «Перед решением», чтобы теория не отодвигала builder на каждом шаге.
